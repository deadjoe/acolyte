# Acolyte内容分析系统设计文档

## 1. 系统概述
Acolyte是一个内容分析评估系统，专注于检测文本内容中的偏见、误导性和隐藏意图。系统支持通过Web界面、CLI和API三种方式提交内容，由单个或多个LLM进行分析评估。系统支持结果聚合、多LLM评议和历史记录查询功能。

系统的核心特色是综合偏见检测框架，包括语言层面分析、叙事结构分析、框架效应分析、事实准确性分析、逻辑谬误分析等多个维度，并提供量化评分系统计算内容的可信度。

## 2. 系统架构
系统采用前后端分离架构，后端使用分层设计：

### 2.1 整体架构
- **API层**：基于FastAPI的RESTful API接口
- **服务层**：处理业务逻辑，与数据库和外部服务交互
- **领域层**：核心业务逻辑和实体模型
- **数据层**：数据存储和访问
- **前端**：基于React、Tailwind CSS和shadcn UI组件的Web界面
- **CLI客户端**：使用Click和Rich实现富文本显示的命令行工具
- **日志系统**：全面的统一日志系统，支持多级别日志记录

### 2.2 分层架构细节
- **API层**：
  - 路由定义与请求处理
  - 请求参数验证
  - 响应格式化
  - 异常处理与统一响应
  - 依赖项注入

- **服务层**：
  - TaskService: 任务创建、处理、查询
  - LlmService: LLM配置管理、测试、使用
  - PromptService: 提示词模板管理
  - ConfigService: 系统配置管理
  - 事务管理与会话控制

- **领域层**：
  - 任务处理器（策略模式）
  - LLM客户端（工厂模式）
  - 提示词管理器
  - 响应解析器
  - 领域事件

- **数据层**：
  - SQLite数据库
  - SQLAlchemy ORM模型
  - 会话管理工具
  - 数据访问接口

#### 3.1.2 Ollama客户端设计
- **基本架构**：
  - 继承LlmClient抽象基类，实现所有必要的抽象方法
  - 支持本地部署的开源模型（Llama、Mistral、Mixtral等）
  - 默认连接到localhost:11434，可配置远程服务器
  - 支持Generate API，计划支持Chat API

- **初始化过程**：
  - 调用父类初始化基本属性
  - 设置提供商信息（PROVIDER_OLLAMA）
  - 标准化基础URL，默认为http://localhost:11434/api
  - 设置更长的超时时间（本地模型可能需要更多时间响应）

- **API调用流程**：
  - 使用Generate API发送请求，支持系统提示词和用户提示词
  - 请求参数包含模型名称、提示词、温度等设置
  - 支持JSON格式输出，提高解析成功率
  - 应用重试机制，处理临时错误

- **错误处理**：
  - 使用@retry_on_error()装饰器增强错误恢复能力
  - 分类处理不同类型的错误（超时、HTTP状态错误、请求错误等）
  - 详细的日志记录，便于诊断问题

- **连接测试**：
  - 实现_test_connection方法，发送轻量级请求测试连接
  - 检查指定的模型是否存在
  - 返回标准化的测试结果

- **文档化**：
  - 详细的类和方法文档字符串
  - 清晰的参数和返回值说明
  - 中文注释，保持与项目风格一致

#### 3.1.1 LLM响应解析架构
- **解析流程**：
  1. LLM客户端调用API获取响应
  2. 客户端的`_parse_response`方法调用特定LLM的解析方法（如`ResponseParser.parse_openai_response`）
  3. 特定LLM解析方法调用通用的基础解析方法`ResponseParser.parse_base_response`
  4. 基础解析方法提取评分和结构化内容
  5. 任务处理器保存解析结果到数据库

- **主要解析方法**：
  - `ResponseParser.parse_base_response`：通用基础解析方法，处理所有LLM共享的解析逻辑
  - `ResponseParser.parse_openai_response`：OpenAI特定的解析方法
  - `ResponseParser.parse_anthropic_response`：Anthropic特定的解析方法
  - `ResponseParser.parse_gemini_response`：Google Gemini特定的解析方法
  - `ResponseParser.parse_deepseek_response`：DeepSeek特定的解析方法
  - `ResponseParser.parse_ollama_response`：Ollama特定的解析方法，用于处理本地模型的响应
  - `ResponseParser.extract_scores`：提取评分指标（BI、MI、HI、CS）
  - `ResponseParser._extract_json_scores`：从JSON结构提取评分（优先策略）
  - `ResponseParser._extract_regex_scores`：使用正则表达式提取评分（备用策略）
  - `ResponseParser.extract_structured_content`：提取结构化内容
  - `ResponseParser.extract_limitations`：提取分析局限项

- **评分提取策略**：
  - 优先使用JSON解析方式提取评分：
    1. 首先定位"6. 量化评分"章节
    2. 尝试从代码块中提取JSON
    3. 如果没有找到代码块中的JSON，尝试直接提取JSON对象
    4. 解析JSON并提取评分数据
    5. 支持多种JSON结构：
       - 嵌套结构：`{"偏见指数": {"分数": 4.0}}`
       - 直接结构：`{"偏见指数": 4.0}`
       - 数组结构：`[
         {"指标": "偏见指数", "分数": 4.0},
         {"指标": "误导性指数", "分数": 5.0}
       ]`
       - 字段名支持中英文混合："bias_index"/"偏见指数"/"BI"

  - 如果JSON解析失败或不完整，使用正则表达式匹配：
    1. 偏见指数(BI)：`(?:偏见指数|Bias Index|BI)[：:]\\s*(\\d+(?:\\.\\d+)?)`
    2. 误导性指数(MI)：`(?:误导性指数|Misleading Index|MI)[：:]\\s*(\\d+(?:\\.\\d+)?)`
    3. 隐藏意图指数(HI)：`(?:隐藏意图指数|Hidden Intent Index|HI)[：:]\\s*(\\d+(?:\\.\\d+)?)`
    4. 综合可信度(CS)：`(?:综合可信度|Credibility Score|CS)[：:]\\s*(\\d+(?:\\.\\d+)?)`
    5. 支持列表项格式：`• 偏见指数 = 7.5`
    6. 支持多种分隔符：冒号、等号、连字符、括号

- **结构化内容提取**：
  - 使用`extract_structured_content`方法提取预期章节的内容
  - 支持Markdown格式和普通文本格式
  - 动态构建正则表达式，匹配章节标题和内容
  - 提取章节内容直到下一个章节或文档结束

- **分析局限性提取**：
  - 使用`extract_limitations`方法提取分析局限性列表
  - 首先查找"7. 分析局限"章节
  - 尝试匹配Markdown列表项
  - 如果没有找到列表项，按行分割文本

- **错误处理与日志记录**：
  - 详细的日志记录，跟踪解析过程的每个步骤
  - 健壮的错误处理，捕获并处理所有可能的异常
  - 提供清晰的错误信息，便于诊断问题
  - 支持不同级别的日志记录（DEBUG、INFO、WARNING、ERROR）

## 4. 会话管理与数据访问

### 4.1 会话管理设计
- SessionManager 类实现的关键功能：
  - session_scope：同步会话上下文管理器
  - async_session_scope：异步会话上下文管理器
  - extract_model_data：从ORM对象提取纯数据
  - with_session：会话函数装饰器（同步）
  - async_with_session：会话函数装饰器（异步）
- 使用模式：
  - API层通过依赖注入获取会话
  - 服务层接收会话并管理事务
  - 在异步环境中防止DetachedInstanceError
  - 错误处理与事务回滚

### 4.2 数据库设计
#### 4.2.1 表结构
- **llm_configs**：LLM配置信息（名称、API密钥、基础URL、模型名称等）
- **prompts**：提示词模板（版本、内容、描述、创建时间等）
- **tasks**：任务记录（ID、提交时间、文本内容、处理方法、状态等）
- **task_results**：任务结果（任务ID、LLM ID、原始响应、处理结果、分数等）
- **task_llms**：任务与LLM的关联（任务ID、LLM ID列表、是否评议者）
- **reviewer_votes**：评议者投票记录（任务ID、评议者ID、投票的结果ID）

#### 4.2.2 数据库关系
- tasks与task_results的一对多关系
- tasks与llm_configs通过task_llms的多对多关系
- prompts与tasks的一对多关系
- llm_configs与task_results的一对多关系

## 5. 提示词管理策略
### 5.1 存储方式
- **文件系统**：维护prompt目录作为主存储，便于版本控制和直接编辑
- **数据库镜像**：将最新版本同步到数据库中，便于API访问
- **版本追踪**：在数据库中记录不同版本，支持历史版本查询和回滚
- **模型特定提示词**：支持为特定LLM模型优化的变体提示词

### 5.2 提示词加载流程
1. 系统启动时扫描prompt目录，获取最新版本
2. 与数据库记录比较，并将更新同步到数据库
3. 运行时优先从数据库中读取，提高访问效率
4. 支持特殊格式检测（如Claude特定的`_v3.md`格式）
5. 记录所有提示词加载和同步操作

## 7. 实现过程
### 7.1 任务处理流程
1. **任务提交**
   - 接收API/CLI提交的内容和处理方法
   - 通过服务层创建任务记录和关联数据
   - 触发异步处理任务

2. **处理器选择**
   - TaskProcessor根据处理模式选择合适的处理器策略：
     - SingleLlmProcessor: 使用单个LLM处理
     - MultipleLlmProcessor: 使用多个LLM并行处理
     - ReviewProcessor: 对多个LLM结果进行评议

3. **处理执行**
   - 获取最新的提示词模板（优先使用模型特定版本）
   - 获取处理所需的LLM客户端
   - 根据处理器策略执行处理：
     - **单LLM处理**：直接调用LLM处理内容
     - **多LLM并行处理**：使用asyncio并行调用多个LLM
     - **评议处理**：
       - 收集所有LLM的结果
       - 单评议者模式：将结果发送给指定评议者
       - 多评议者模式：多个评议者投票，选择最多票的结果

4. **结果处理**
   - 解析LLM响应，提取偏见指数等关键数据
   - 转换为标准化结果格式
   - 保存结果到数据库
   - 更新任务状态

5. **异常处理与恢复**
   - 针对临时错误自动重试（如API速率限制）
   - 针对持久错误记录详细日志
   - 更新任务为失败状态并提供失败原因
   - 服务层提供事务一致性保证

### 7.2 结果呈现策略
- **Web界面**：美观格式化文本+图形化分数显示
- **CLI**：使用Rich库的简洁文本格式，整齐布局
- **API**：标准JSON响应，结构一致的数据
- **日志**：详细的系统日志，用于诊断和监控

### 7.3 错误处理与重试策略
#### 7.3.1 多层错误处理
1. **客户端层错误处理**
   - LLM客户端使用统一的ErrorInfo模型表示错误
   - 对不同API提供商的特定错误进行标准化处理
   - 所有HTTP请求包含完整的错误捕获和处理
   - 将原始错误信息转换为结构化错误，便于解析

2. **服务层错误处理**
   - 服务方法包含完整的try-except块
   - 事务操作保证数据一致性
   - 异常转换为统一的返回格式，包含success标志和错误信息
   - 会话管理工具确保资源正确释放

3. **API层错误处理**
   - FastAPI异常处理中间件捕获未处理异常
   - 使用适当的HTTP状态码响应错误
   - 转换服务层错误到标准HTTP响应
   - 提供详细的错误消息和可操作建议

#### 7.3.2 重试机制
1. **自适应退避算法**
   - 使用指数退避策略，随重试次数增加等待时间
   - 对不同错误类型采用不同重试策略
   - 尊重API提供商的Retry-After头信息
   - 添加随机抖动避免请求同步

2. **可重试错误识别**
   - 速率限制(429)错误自动重试
   - 服务端临时错误(5xx)自动重试
   - 网络连接错误和超时自动重试
   - 无法恢复的错误(401认证失败等)不重试

3. **重试配置**
   - 每个LLM提供商可定制重试参数
   - 可设置最大重试次数、初始延迟和最大延迟
   - 请求超时可配置，针对不同API优化
   - 通过日志记录所有重试尝试和结果

#### 7.3.3 错误报告与恢复
1. **结构化日志记录**
   - 错误发生时记录完整上下文
   - 使用特定错误代码便于分析和聚合
   - 记录重试次数、时间间隔和结果
   - 长期错误模式监控和分析

2. **任务状态恢复**
   - 记录任务失败原因和上下文
   - 提供任务重新提交机制
   - 实现部分结果保存，支持部分失败场景
   - 报告具体哪个LLM失败及原因

- 在`utils/logging.py`中集中配置日志
- 环境变量控制：`ACOLYTE_LOG_LEVEL`、`ACOLYTE_LOG_TO_FILE`、`ACOLYTE_LOG_DIR`
- 支持多种输出目的地（控制台、文件）
- 不同输出使用不同日志格式（控制台简化版、文件详细版）

### 9.2 日志级别
- **DEBUG**：详细的开发和诊断信息
- **INFO**：一般操作事件和里程碑
- **WARNING**：不影响操作的潜在问题
- **ERROR**：允许应用程序继续运行的错误
- **CRITICAL**：可能导致应用程序故障的严重错误

### 9.3 模块特定日志
- API请求/响应日志
- 数据库事务日志
- LLM客户端交互日志
- 任务处理工作流日志
- 配置管理日志
- CLI命令执行日志

## 10. 架构模式与技术基础设施
系统使用多种架构和设计模式以提高可维护性和扩展性：

### 10.1 分层架构
- 将系统分为API层、服务层、领域层和数据层
- 每层具有明确的职责和边界
- 依赖关系始终从上层指向下层
- 确保业务逻辑与接口实现分离

### 10.2 服务层模式
- 将业务逻辑从API控制器中分离
- 服务封装了完整的业务操作
- 提高了代码的可测试性和复用性
- 标准化返回格式和错误处理

### 10.3 策略模式
- 任务处理器使用策略模式实现不同处理策略
- BaseTaskProcessor定义通用接口
- 具体处理器实现不同的处理策略（Single/Multiple/Review）
- 在运行时动态选择适当的处理器

### 10.4 工厂模式
- LLM客户端使用工厂模式创建
- get_client_for_llm函数根据配置动态创建适当的客户端
- 支持添加新的LLM提供商而无需修改使用代码
- 智能提供商检测算法自动选择正确的客户端

### 10.5 依赖注入
- API路由使用依赖注入获取所需服务和资源
- SessionManager提供会话注入
- 提高了代码的可测试性和组件解耦
- 简化单元测试中的依赖模拟

### 10.6 单一职责原则
- 每个组件和类都有明确的单一职责
- ResponseParser专注于响应解析
- ErrorHandler专注于错误处理
- SessionManager专注于会话管理

### 10.7 会话管理框架
#### 10.7.1 会话管理器设计
- **SessionManager类**：提供会话管理核心功能
- **上下文管理器**：同步和异步会话上下文支持
- **ORM数据提取**：解决会话外访问ORM对象问题
- **事务封装**：统一的事务控制和异常处理

#### 10.7.2 会话使用模式
- **函数装饰器**：简化会话获取和传递
- **依赖注入**：在API路由中获取会话
- **数据分离**：将ORM对象转换为纯数据字典
- **预加载关系**：使用join语句预加载关系避免懒加载问题

#### 10.7.3 异步环境中的会话管理
- **async会话**：支持异步环境中的会话操作
- **并发控制**：确保会话线程安全性
- **生命周期管理**：确保会话资源正确释放
- **错误恢复**：异常情况下的会话清理和回滚

### 10.8 HTTP客户端框架
#### 10.8.1 客户端管理器
- **HttpClientManager**：管理HTTP客户端池
- **客户端配置**：为不同提供商定制连接选项
- **连接池**：优化连接资源利用
- **资源清理**：确保应用程序退出时连接正确关闭

#### 10.8.2 异步HTTP请求
- **基于HTTPX**：使用现代异步HTTP客户端库
- **重试机制**：内置重试逻辑处理临时故障
- **超时控制**：不同阶段的超时设置（连接、读取）
- **日志记录**：详细的请求和响应日志

#### 10.8.3 安全性考虑
- **敏感信息处理**：日志中自动隐藏API密钥
- **HTTPS验证**：默认验证SSL证书
- **超时保护**：防止长时间挂起的请求
- **请求限制**：控制并发连接数
